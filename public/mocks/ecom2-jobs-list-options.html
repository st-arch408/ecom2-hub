Jobs List — Layout Options

Three interaction patterns for the Jobs List screen (JL-01). The Jobs List is the central hub of the platform's "job-centric" identity — every property address Marcus works on lives here. The right pattern needs to balance scannability across 12+ jobs with rich, contextual detail for each one.

Screen ID: JL-01 Date: March 5, 2026 Status: UX Decision Required Persona: Marcus Rivera, Tier 3 Solo Operator
Option A
Property Cards → Separate Detail Page
Each job is a visual card grouped by property address. Cards show aggregate status, progress, and key metrics. Clicking navigates to a full Job Detail page with tabbed sub-views (Proposals, Orders, Deliveries, Margin). Clean level separation — list view is purely for scanning and selection.
+ Visual, scannable, feels job-centric
+ Room for upsell/cross-sell tags
⚠ Requires page navigation to see detail
⚠ Cards take more vertical space

Expandable Cards — Inline Detail
Compact card rows with key data visible at a glance. Clicking a card expands it in-place to reveal a detail panel with proposal status, order progress, delivery info, and margin data. No page navigation needed for quick checks. Full Job Detail page still available via "View Full Detail" button.
+ Quick access to detail without navigation
+ Compact list for scanning
⚠ Can get crowded with multiple expanded
⚠ Limited space for complex detail

Option C
Master-Detail Split Panel
Persistent list on the left, detail panel on the right. Clicking a job in the list instantly loads its full context — proposals, orders, deliveries, margin data, and upsell opportunities — without page navigation. The list stays visible so Marcus can quickly jump between jobs. Mirrors the email client pattern (Outlook, Gmail) that contractors already know.
+ Instant context switching, no page loads
+ Rich detail always visible
⚠ List panel is narrow — less data per row

Design Considerations
All options share: Status filter tabs, + New Job action, same status badge system, same sidebar/topbar chrome, same persona data (Marcus Rivera, Rivera Roofing).

Mobile consideration: Options A and B adapt naturally to single-column. Option C requires collapsing the split panel into a list → detail navigation on mobile (similar to iOS Mail).

Job entity alignment: All three options treat the Job as a property-level container (per the Platform Overview doc). The "job" is the address, not a single order. Multiple proposals, orders, and deliveries can live under one Job.

Cross-sell signals: All options surface the platform's future multi-trade upsell capability. Completed roofing jobs with detected siding, painting, or gutter opportunities show a purple "opportunity" indicator.

Decision needed: Which pattern becomes the standard for all list views (Jobs, Proposals, Deliveries, Invoices)? Or should different sections use different patterns based on information density?
